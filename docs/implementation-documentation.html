1. Executive Summary

MyRota is a React and Vite based rota management application that now opens directly into a live operational dashboard instead of a login landing page. The application supports user-facing rota visibility, admin-controlled rota editing, live team status and leave planning, ONCALL reporting with Excel export, and a Jira dashboard backed by a Netlify Function.

Firestore remains the source of truth for users, rota assignments, admin configuration, and audit logs. The Jira integration was intentionally moved away from Firebase Functions because production Firebase Functions required a paid plan. The current production Jira proxy runs on Netlify Functions and supports a mock mode for preview and branch deploys.

Business Outcome

  • Home page shows live working status and next three days of leave planning.
  • Users can view rota data but only admin can modify shifts or leave.
  • Jira issue counts and details are visible directly inside the app.
  • ONCALL / night / weekend shift reporting can be exported to Excel.

Key Technical Outcome

  • Netlify now hosts both the frontend and the Jira backend proxy.
  • Firestore user ordering follows the numeric position field.
  • Browser Firestore reliability was improved with long-polling configuration.
  • Preview environments can show mock Jira data without using live credentials.
Current production state: the live Jira endpoint /.netlify/functions/jiraDashboard?mode=summary was validated on April 4, 2026 and returned live counts successfully.

4. Delivered Functional Scope

4.1 Home Page

  • The application opens directly into the home dashboard instead of a login landing banner.
  • The main title is What about Today?.
  • Two glassmorphism panels are shown side by side on large screens and stacked on mobile.
  • The live status panel uses green glass styling.
  • The leave plan panel uses red / rose glass styling.
  • Below those two panels, the Jira dashboard appears inside a yellow glass section.

4.2 Live Team Status Panel

  • Shows the current date in IST.
  • Displays who is on leave today.
  • Displays who is doing morning shift today.
  • Displays who is doing night shift today.
  • On weekends, shows weekend support availability in the simplified format {name} is available on weekend.
  • Avoids duplicate weekend notifications.

4.3 Leave Plan Panel

  • Shows leave information for the next three days.
  • Uses relative wording such as tomorrow and day after tomorrow.
  • Falls back to a positive empty state when no leave is planned.

4.4 ROTA Dashboard

  • Opens from the ROTA button in the top navigation.
  • The ROTA button is visually highlighted and marked with a NEW ribbon.
  • Supports week tabs plus a whole-month view.
  • Supports month and year selection.
  • The year selector is dynamic and allows the next year once the current month is December.
  • Tables use a readable sticky employee column on mobile.

4.5 ONCALL Report

  • The old dialer icon was replaced by a labeled ONCALL button.
  • The ONCALL page lists C and WS assignments for the selected month.
  • The resulting report can be exported to Excel.

4.6 Jira Dashboard

  • Shows EDI_MAPPING, EDI_SUPPORT, and CHANGE_REQUEST sections.
  • Each count opens a separate Jira detail page inside the application.
  • The details page links out to the underlying Jira issue.
  • Refresh actions are available on both summary and detail views.
  • Preview environments can run with mock Jira data instead of live credentials.

5. Access Control and User Experience Rules

5.1 Admin Mode

  • Admin mode is entered by clicking the MyRota logo.
  • When already in admin mode, clicking the logo returns the user to home.
  • The admin password is read from users/admin first and falls back to config/admin.
  • Only admin users can change rota data or see the change log page.

5.2 User Mode Restrictions

  • Regular users have view-only access.
  • The user-facing leave update path was removed.
  • The employee document with ID admin is hidden from user-facing employee lists and notifications.

5.3 Employee List Rules

  • Users are read from Firestore collection users.
  • Display order follows the numeric position field.
  • Missing or invalid positions sort to the bottom.
  • The display name replacement rule maps Asutosh to Komal.
  • Duplicate names are removed after ordering.

5.4 Mobile UX Updates

  • The mobile logo size was reduced to prevent overlap with the ROTA button.
  • The week table preserves horizontal scroll position while editing.
  • The sticky name column keeps employee names visible while scrolling horizontally.

6. Business Logic Rules

6.1 Shift and Leave Codes

A = Morning Shift B = Normal Shift C = Night Shift WS = Weekend Shift W = Weekend Off PL = Personal Leave RH = Restricted Holiday CH = Company Holiday HD = Half Day

6.2 Calendar Rules

  • Base year is 2025.
  • For 2025, only November and December are available.
  • All years after 2025 expose the full January to December range.
  • When the current month is December, the next year is added to the selector.

6.3 Report Rules

  • The ONCALL report includes only rows where the effective shift is C or WS.
  • Report rows are built from the selected month, selected year, employee list, and final resolved shift values.

6.4 Rota Value Resolution

  • Assignments support both name-based and index-based rota keys.
  • Object values may carry both shift and leave.
  • Fallback scanning is used for leave detection where old key shapes or older data formats still exist.

8. Jira Integration

8.1 Current Runtime Model

  • The browser never calls Jira directly.
  • The frontend calls /.netlify/functions/jiraDashboard.
  • The Netlify Function reads JIRA_BASE_URL, JIRA_USERNAME, and JIRA_PASSWORD from environment variables.
  • The backend performs Jira REST API search requests using Basic authentication.

8.2 Why Netlify Functions Were Introduced

  • Firebase Functions deployment required paid billing enablement for the project.
  • The application was already hosted on Netlify, making Netlify Functions the simplest free production path.
  • Netlify deploy contexts also made it easy to introduce a test / preview mode.

8.4 Preview / Test Environment Strategy

  • netlify.toml sets JIRA_DASHBOARD_USE_MOCK=true for preview and branch deploys.
  • Production keeps JIRA_DASHBOARD_USE_MOCK=false.
  • The frontend shows Preview instead of Live whenever mock data is active.
  • Mock summary and detail payloads are maintained in netlify/jira/mockData.js.
Production environment variables required in Netlify: JIRA_BASE_URL, JIRA_USERNAME, JIRA_PASSWORD.
