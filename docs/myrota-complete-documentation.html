1. Product Overview

Purpose

MyRota is the central team dashboard for rota management and daily operational visibility. It combines rota views, leave visibility, admin-controlled editing, audit logs, Jira monitoring, and quick navigation to related microsites.

Main User Groups

  • Portal users who need view access to operational information
  • Admin users who maintain rota data and review logs
  • Jira consumers who need issue counts and drill-down lists

Frontend Stack

React 19, Vite, custom CSS, lazy-loaded Jira screens.

Data Layer

Firestore for users, rota data, admin credentials, and audit logs.

Integration Layer

Netlify Functions proxy Jira API access using server-side environment variables.

The application has two separate access concepts: the landing-page portal login for entering the app, and the Edit ROTA admin login used from inside the app when the MyRota logo is clicked.

2. Runtime Architecture

The main application is a client-rendered React single-page app. Firestore handles real-time operational data. Jira data is fetched through a Netlify serverless function so Jira credentials remain off the client.

Jira is intentionally not called directly from the browser. The Netlify function protects Jira credentials and standardizes request and response handling.

APF/ Microsites and legacy site assets Published selectively but not part of the main React runtime path

4. Firestore Data Model

Path Data Stored Usage In App
users/{username} Employee metadata and sort position Used to build and order the employee list shown in the rota views
users/admin Admin password field Primary source for Edit ROTA admin login verification
rota/master Central keyed rota map Stores shift and leave assignments for each day and employee slot
logs/{logId} Audit history Stores admin edits and supports the Logs page
config/admin Optional backup admin settings Secondary fallback if `users/admin` is not available
The current admin login path for Edit ROTA is designed to read the password from `users/admin` first, which aligns with the live Firestore document structure.

5. End-To-End Flows

Portal Login Flow

  • User lands on the portal login screen.
  • App validates the default portal credentials stored in the frontend constants.
  • Session state is written into browser storage.
  • User is taken to the home page and navigation is unlocked.

Edit ROTA Admin Flow

  • User clicks the MyRota logo from inside the app.
  • Admin modal opens and requests the rota edit password.
  • App fetches `users/admin` from Firestore.
  • Password match enables admin state, rota editing, and logs access.

Jira Summary Flow

  • Home page reaches the Jira section.
  • Frontend calls the Netlify Jira function in summary mode.
  • Function runs predefined JQL queries and returns normalized counts.
  • User sees counters and recent issues.

Jira Detail Flow

  • User clicks a Jira card or queue.
  • Frontend calls the same function in detail mode with `view` id.
  • Function returns full issue rows for that specific JQL view.
  • User can open the issue directly in Jira.

6. Build And Deployment

Step 1

`npm run build` produces the main Vite frontend output in `dist/`.

Step 2

`scripts/copy-microsites.mjs` copies selected APF, SFTP, Certificate, and Documentation assets into `dist/`.

Step 3

Netlify serves `dist/` and hosts the Jira serverless endpoint from `netlify/functions`.

System Configuration Purpose
Netlify `publish = "dist"` and `functions = "netlify/functions"` Hosts frontend and serverless endpoints
Jira Function `JIRA_BASE_URL`, `JIRA_USERNAME`, `JIRA_PASSWORD` Allows secure server-side Jira access
Preview behavior `JIRA_DASHBOARD_USE_MOCK` Enables mock Jira data for preview or branch deployments

7. Security, Operations, And Maintenance

Operational Notes

  • Portal login is browser-side and separate from Edit ROTA admin mode.
  • Admin edit access depends on the `users/admin` Firestore document.
  • Rota edits are logged into `logs/*` for traceability.
  • Certificate reminders are stored locally in browser storage, not Firestore.

Recommended Maintenance Boundary

  • Main ownership zone: `src/`, `src/jira/`, `netlify/`, `scripts/`, `docs/`
  • Change APF folders only when those microsites are intentionally updated.
  • Keep generated assets and unrelated legacy folders out of routine cleanup work.
  • Validate with a production build after each meaningful change.
The portal login credentials are still defined inside frontend code. For stronger security, the next improvement would be moving that gate to a server-backed or identity-backed model.
